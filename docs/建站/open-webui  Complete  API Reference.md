Conventions</h2><p dir="auto" data-imt-p="1">The OpenWebUI API is organized into logical prefixes that separate different functional concerns:</p><p dir="auto" data-imt-p="1"><strong>Primary Management APIs (<code class="notranslate">/api</code>)</strong>: All management APIs that drive the user interface live under this prefix, including chat session management, model configuration, and administrative functions.</p><p dir="auto" data-imt-p="1"><strong>File &amp; RAG Management (<code class="notranslate">/api/v1</code>)</strong>: File upload and knowledge-base Retrieval-Augmented Generation endpoints are mounted under this dedicated namespace to handle document processing and vector indexing.</p><p dir="auto" data-imt-p="1"><strong>OpenAI Compatibility (<code class="notranslate">/v1</code>)</strong>: A separate stateless OpenAI-compatible layer is mounted at this prefix for external integrations expecting standard OpenAI API semantics.</p><p dir="auto" data-imt-p="1"><strong>Ollama Integration (<code class="notranslate">/ollama</code>)</strong>: Native Ollama pass-through endpoints are mounted under this prefix for direct access to Ollama's native API functionality.</p><p dir="auto" data-imt-p="1">All requests requiring authentication must include a valid bearer token in the Authorization header using the format: <code class="notranslate">Authorization: Bearer &lt;token&gt;</code></p><h2 dir="auto" data-imt-p="1">Authentication &amp; Error Handling</h2><h3 dir="auto">Authentication Requirements</h3><p dir="auto" data-imt-p="1">Authentication is enforced through FastAPI dependencies using the <code class="notranslate">get_verified_user</code> function. All endpoints requiring authentication will return <code class="notranslate">401 Unauthorized</code> for missing or invalid tokens. Authorization and permissions are enforced at both model and administrative levels, with insufficient permissions yielding <code class="notranslate">403 Forbidden</code> responses.</p><p dir="auto" data-imt-p="1">Bearer tokens can be generated and managed through the web interface under Settings &gt; Account, and must be included in all authenticated requests.</p><h3 dir="auto">Common HTTP Status Codes</h3><p dir="auto" data-imt-p="1">The API follows standard HTTP status code conventions:</p><p dir="auto" data-imt-p="1"><strong>Success Responses</strong>: <code class="notranslate">200 OK</code> indicates successful requests, <code class="notranslate">201 Created</code> signifies successful resource creation such as new chats, and <code class="notranslate">204 No Content</code> confirms successful deletion operations.</p><p dir="auto" data-imt-p="1"><strong>Client Error Responses</strong>: <code class="notranslate">400 Bad Request</code> and <code class="notranslate">422 Unprocessable Entity</code> indicate malformed payloads, <code class="notranslate">401 Unauthorized</code> signals missing or invalid authentication tokens, <code class="notranslate">403 Forbidden</code> represents insufficient permissions, and <code class="notranslate">404 Not Found</code> indicates missing resources.</p><p dir="auto" data-imt-p="1"><strong>Server Error Responses</strong>: <code class="notranslate">5xx</code> status codes represent server errors, typically from upstream model adapter failures or internal server faults.</p><h3 dir="auto">Error Response Format</h3><p dir="auto" data-imt-p="1">Typical error payloads follow a consistent structure:</p><div class="highlight highlight-source-json notranslate position-relative overflow-auto" dir="auto"><pre class="notranslate" style="font-family: &quot;Monaspace Neon&quot;, ui-monospace, SFMono-Regular, &quot;SF Mono&quot;, Menlo, Consolas, &quot;Liberation Mono&quot;, monospace;">{

The endpoint operates statelessly unless clients explicitly associate requests with a <code class="notranslate">chat_id</code> parameter.</p><h3 dir="auto">Request Processing Flow</h3><p dir="auto"
